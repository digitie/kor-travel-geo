### 두 에이전트 병행 권장 순서

두 에이전트가 동시에 진행할 때는 **Agent A = Codex(데이터 원천 보강·검증 owner)**, **Agent B = Claude Code(source registry/upload/reconcile/match-set 기반 owner)** 로 나눈다. Agent B는 `T-206`을 `T-118` 확정 전에는 착수하지 않는다. `T-200`은 B 전 체인과 A의 `T-206` seed까지의 단일 root이므로 가장 먼저 머지해 모두의 대기를 푼다.

병행 운영 원칙 (**두 Agent가 병행하며 서로의 작업을 반드시 상호 리뷰한다** — 아래 PR·리뷰 루프를 두 Agent 모두 동일하게 따른다):

- **모든 Task는 PR 후 머지한다.** main 직접 commit 금지. 작업은 `agent/<a|b>-<...>` branch에서 하고 CI(backend/frontend/openapi) green 확인 후 squash 머지한다.
- **Task 착수 전(필수 선행) — 과거 모든 PR 리뷰 스캔:** 직전 PR만이 아니라 **과거 전체 PR(open/closed 무관, 자기 것·상대 것 모두)**을 검색해, 다른 Agent가 남긴 **아직 반영·답변하지 않은 리뷰 코멘트/리뷰 thread**가 있는지 확인한다. 있으면 그 리뷰를 **먼저 반영(또는 답변) → PR → 머지**한 뒤 새 Task를 시작한다. (한 번 처리한 리뷰는 답변/반영 코멘트로 종결해 다음 스캔에서 중복 처리하지 않는다.)
- **Task 머지 후 — 대기하지 않는다:** 자신의 Task PR을 머지한 뒤 상대 Agent의 리뷰를 **기다리지 않는다.** 리뷰가 없으면 그대로 다음 단계로 진행하고(필요하면 그대로 PR → 머지), 상대 리뷰는 위 "과거 모든 PR 리뷰 스캔"에서 **발견하는 즉시** 반영 → PR → 머지로 처리한다. (이전의 "리뷰가 달릴 때까지 대기" 규칙은 폐기.)
- **Task 머지 후(필수 후행) — 상대 PR 상세 리뷰:** 다음 Task로 넘어가기 전, **과거 전체 PR(open/closed 무관)** 중 다른 Agent가 올렸고 자신이 아직 리뷰하지 않은 PR에 **상세 리뷰 코멘트**(H/M/L 분류 + 잘된 점 + 머지조건/후속 권장)를 남긴다. 반영할 게 없으면 표준 종결 코멘트(예: "사후 리뷰 완료 — 블로킹 0 / 반영 불필요")로 닫는다. **단, 리뷰 반영(fixup) PR — 즉 다른 Agent의 리뷰 코멘트를 코드로 반영한 PR(`*review*fixup*`, `*리뷰*후속*` 류) — 은 추가 리뷰하지 않는다**(리뷰→fixup→fixup의 리뷰… 무한 루프 방지). fixup PR은 그대로 머지하고, 거기서 새 결함이 보이면 별도 Task/이슈로 만든다.
- **작업 중 작업량이 늘어나면** 즉흥 확장하지 말고 **GitHub 이슈를 먼저 만들어** 범위를 적고 → 작업 → PR → 머지한다(이슈 → 작업 → PR → 머지).
- 작업 중에는 주기적으로 `origin/main`과 현재 작업 branch의 최신 상태를 확인하고, 장기 작업은 중간에 rebase 또는 fast-forward로 충돌을 작게 유지한다.
- Git 작업은 Windows Git과 NTFS worktree에서 수행하고, 그 외 의존성 설치·테스트·장기 실행 검증은 WSL ext4 테스트 미러에서 수행한다.
- 작업 중 반복적으로 문제가 되는 패턴은 `docs/agent-failure-patterns.md` 또는 관련 workflow 문서에 기록해 같은 실패가 반복되지 않게 한다.

- T-219 v1 VWorld 계약 정합·검증 보강 (T-106 #189 사후 리뷰 후속) — T-106 리뷰(PR #189 코멘트, 다중 에이전트 적대적 리뷰: 확정 H0/M5/L5)에서 나온 후속 부채를 정리한다. 런타임 wire 동작은 정확하므로 범위는 **published 계약(OpenAPI/생성타입) 정합**과 **테스트 커버리지**, 그리고 v2 side-effect 결정이다.
  - **M1(근본)**: 라우터가 raw `ORJSONResponse`(`vworld_success_response`) 반환 → `response_model`/`response_model_exclude_none`이 문서 전용·무효, 실 payload와 두 경로로 갈림. 생성 openapi.json 스키마에 대표 런타임 body(default/geocode simple/refine=false/reverse simple/NOT_FOUND 200/도메인 에러)를 검증하는 계약·회귀 테스트를 추가해 drift가 CI에서 깨지게 한다(이 한 건이 M2·M3·M5 상당수를 동시에 닫음).
  - **M2**: openapi.json/생성타입이 `simple/refine` 생략 필드(`input`/`refined`/`result[].type`)를 `required`+`additionalProperties:false`로 표기 → optional화(또는 success-variant 분리) 후 재생성. **wire 동작은 VWorld-correct이므로 변경 금지.**
  - **M3**: 새 400 VWorld error envelope(`VWorldErrorEnvelope/Body/Detail`)이 어떤 라우트에도 연결되지 않아 스펙에 없음(v1 geocode/reverse는 200+422만 광고). `responses={400: {"model": VWorldErrorEnvelope}}`(+429/5xx) 연결, 오해 소지 422 억제, openapi.json+api.gen.ts+schemas.gen.ts 재생성.
