# T-027/T-047 국가지점번호 포함 재적재·튜닝 재측정

## 요약

2026-05-31 밤부터 2026-06-01 새벽까지 T-027 전체 적재와 T-047 쿼리 성능 측정을 다시 수행했다. 이번 run의 목적은 다음 세 가지였다.

1. 국가지점번호(`TL_SPPN_MAKAREA`)까지 포함한 클린 전체 적재를 다시 검증한다.
2. T-047 benchmark에서 국가지점번호 geocode와 reverse 경로를 분리해 측정한다.
3. 전체 daily ZIP을 실제 데이터로 증분 적용하고, 데이터가 보강되면 더 좋아질 부분을 문서화한다.

주요 결론:

- 새 Docker DB `kor-travel-geo-t027-retune`를 port `15435`에 띄워 기존 T-027 최종 DB(`15434`)는 보존했다.
- 전국 full-load, direct 출입구, 국가지점번호, 20260401 daily 적용, 링크 해소, MV swap까지 완료했다.
- 남은 daily ZIP 35개(`20260402`~`20260506`)도 실제 데이터로 모두 적용했다. 실제 daily 데이터가 충분했으므로 임의 synthetic delta는 만들지 않았다.
- 최종 row count는 `mv_geocode_target=6,418,735`, `mv_geocode_text_search=6,418,735`, `tl_sppn_makarea=24,204`다.
- T-047 SQL benchmark는 2,000 case, 18,000 measurement, error 0으로 완료했다. Q11 국가지점번호는 `sppn_geocode`와 `sppn_reverse`를 각각 측정했고, concurrency 64 p95가 각각 `90.22ms`, `87.45ms`였다.
- DB SQL 자체는 새 index 없이도 목표 범위에 있어 추가 SQL 튜닝은 하지 않았다. 대신 benchmark blind spot, v2 reverse 응답 승격, full-load smoke script의 v2 client 계약 불일치를 보정했다.

## 이번 PR의 코드 개선

| 변경 | 전 | 후 |
|------|----|----|
| benchmark Q11 coverage | `sppn_reverse`만 측정 | `sppn_geocode`와 `sppn_reverse`를 모두 측정 |

새 SQL index나 helper MV는 추가하지 않았다. Q11은 `pg_stat_statements` mean execution이 0.5ms 이하였고, client p95도 concurrency 64에서 90ms 안팎이었다. 병목은 오히려 Q4 fuzzy search와 Q3 fuzzy wide 경로에 남아 있다.

## 데이터가 보강되면 더 좋아질 부분

이번 최종 DB는 실제로 조회 가능하지만, 원천 기준월이 섞여 있어 정합성 경고와 일부 거리 오류가 남는다. 다음 데이터가 보강되면 품질이 더 좋아질 가능성이 크다.

1. 모든 원천의 기준월을 맞춘다. `tl_juso_text`, `tl_locsum_entrc`, `tl_navi_*`, SHP, direct 출입구, 국가지점번호가 같은 `YYYYMM`이면 C10과 direct 출입구 거리 경고를 더 정확히 해석할 수 있다.
2. daily가 텍스트와 지번 연결만이 아니라 SHP, 위치정보요약DB, 내비게이션용DB, direct 출입구에도 같은 날짜 범위로 제공되면 C4/C6/C7/C8의 기준월 불일치가 줄어든다.
3. epost `zipcode_full.zip`과 `bulk_delivery.csv` 실제 원천을 채우면 현재 0행인 `postal_pobox`, `postal_bulk_delivery` 경로까지 운영 smoke와 benchmark에 포함할 수 있다.
4. SHP `rds_sig_cd` 결측 581건은 로더에서 해결할 수 없는 원천 natural-key 결측이다. 제공자 보정본 또는 별도 보강 테이블이 있으면 C2 `missing_resolve_key`가 줄어든다.
5. C4 500m 초과 중 경도 약 2도 이동 패턴처럼 원천 좌표 이상치가 반복된다. 제공자 보정본을 받거나, 운영에서 신뢰도/blacklist 메타데이터를 따로 두면 reverse 품질을 더 보수적으로 표시할 수 있다.
6. 현재 v2 candidate는 `point_precision`과 `metadata`로만 보조 정보를 담는다. 장기적으로는 원천별 `source_yyyymm`, 좌표 source, same-month 여부를 후보 설명 metadata에 일관되게 넣으면 디버그 UI와 운영 판정이 쉬워진다.
